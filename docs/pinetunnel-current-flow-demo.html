Phase-by-Phase Walk-through

Phase 1 - Install automated

$ pip install pinetunnel Collecting pinetunnel... Installing collected packages: pinetunnel-7.3.2 Successfully installed pinetunnel
What works: Single PyPI package. No Docker, no manual deps beyond Redis. Console script pinetunnel registered.

Phase 2 - Setup wizard automated

User runs pinetunnel with no args. On first run, auto-launches the 11-step setup (in cmd_setup, lines 2241-2411 of apps/cli/main.py).

Problem: The setup finishes "successfully" but the server is running on a raw IP http://203.0.113.42:8000. TradingView webhooks require HTTPS. User must immediately run another command for Cloudflare. Redis is also flagged WARN - user has to start it manually elsewhere.

What works: Secrets auto-generated (32-48 chars), .env chmod 600, migrations run, daemon started, health verified. Solid plumbing.

Phase 3 - Redis manual

Redis is required for WebSocket state, pub/sub, and rate limiting. The CLI detects it but does not install or start it.

$ sudo apt install redis-server && sudo systemctl enable --now redis-server # OR on macOS: $ brew install redis && brew services start redis
Problem: Setup wizard prints [WARN] Redis is not running but continues anyway. Server starts without WS support. User may not realize their WebSocket push to the EA is broken until alerts fail to fire.

Phase 4 - HTTPS via Cloudflare automated

Three sub-flows in cmd_setup_cloudflare (line 1947): A) DNS + API token, B) quick tunnel (instant, temp URL), C) remotely-managed tunnel (paste token from dashboard).

$ pinetunnel setup-cloudflare Cloudflare Setup Options: A) DNS setup (domain + API token, creates pinetunnel.<domain> A record) B) Quick tunnel (no domain needed, instant HTTPS, temporary URL) C) Remotely-managed tunnel (create on Cloudflare dashboard, paste token) Choose A/B/C [C]: b [OK] Quick tunnel started: https://random-words-xxxx.trycloudflare.com [OK] SERVER_BASE_URL updated in .env [OK] Webhook URL: https://random-words-xxxx.trycloudflare.com/webhook
Problem: Quick tunnel URLs are random and change on every restart - useless for production webhooks. Real users need flow A or C, both require manual Cloudflare dashboard work before running the command.
Problem: This is the 2nd command the user runs. After Phase 2 said "Setup complete!" - misleading. The user thinks they're done, then realizes they need HTTPS.

Phase 5 - Telegram bot manual

Telegram is mandatory per the README, but setup does not pause for it. User must leave the terminal, open Telegram, talk to @BotFather, create a bot, get a token, find their user ID via @userinfobot, then edit .env by hand.

# On phone or desktop Telegram app: 1. Open @BotFather 2. /newbot -> pick name -> get token like 7812345678:AAH...token 3. Open @userinfobot -> get your numeric user ID 4. Back in terminal: $ nano .env TELEGRAM_BOT_TOKEN=7812345678:AAH...token TELEGRAM_ADMIN_IDS=123456789 $ pinetunnel stop && pinetunnel start --daemon
Problem: Mandatory secret is collected outside the wizard. User must restart the server by hand. There is no validation - if the token is malformed, the bot silently fails on startup. No way to test the bot from the CLI.

Phase 6 - EA install manual

pinetunnel install-ea (1555 lines in apps/cli/ea_install.py) auto-detects MetaTrader installs on the local Windows machine OR copies over SSH to a remote Windows VPS. Still requires the user to:

  • Have a Windows machine / VPS with MetaTrader 4 or 5 installed
  • Attach the EA to a chart in MetaTrader (manual GUI step)
  • Enable AutoTrading and AlgoTrading in the terminal
  • Enter the license key + secret in the EA input dialog
Problem: The CLI can copy files but cannot attach the EA to a chart. That is a GUI step in MetaTrader that the user must do themselves. No screenshot or guided walkthrough is provided by the CLI - user must read docs/EA_SETUP.md separately.

Phase 7 - TradingView alerts manual

User must go to TradingView, open a chart, create an alert, switch the "Webhook URL" notification to the Cloudflare URL from Phase 4, and paste their alert message in PineScript CSV format.

# In TradingView UI: 1. Open chart 2. Create Alert 3. Notifications -> Webhook URL 4. Paste: https://pinetunnel.example.com/webhook 5. Message body (CSV): KEY,buy,SYMBOL=lots=0.1,sl=100,tp=200,secret=... 6. Save
Problem: User has to leave the terminal, go to TradingView, configure the alert by hand, and remember the CSV syntax. The CLI has a pinetunnel test command to fire a fake webhook, but the user must discover this themselves.

What the CLI Does Well (Keep These)

Auto-secret generation. 32-48 char secrets, chmod 600 on .env, encryption key auto-generated. No user input needed for security.
Graceful color degradation. Respects NO_COLOR, detects TTY, enables VT processing on Windows 10+. Cross-platform.
First-run auto-launch. Bare pinetunnel on first run launches the wizard. No "run this command" friction.
Idempotent migrations. Alembic runs via Python API to avoid shadow-package bugs. Safe to re-run.
Free-port auto-detection. If 8000 is busy, finds 8001+ and updates .env. No manual port juggling.
Dual OS service install. systemd / launchd / sc.exe handled in apps/cli/service.py (442 lines). One command for boot-time autostart.

Observation: 15 commands. A new user needs to learn 6-7 of them to reach a working bridge. The 2869-line file mixes wizard logic, OS service management, Cloudflare, Nginx, EA install, and health checks in one module.

Key Takeaways

  1. 7 phases, 4 manual context switches. Each switch (Redis, Telegram, EA GUI, TradingView) is a place users drop off.
  2. "Setup complete" lies. 3 more commands are needed. The wizard should either do all of it or honestly say "Phase 1 of 4 done."
  3. Server starts before HTTPS. Brief plaintext exposure; SERVER_BASE_URL is wrong until Cloudflare runs.
  4. Redis is silently optional. Wizard warns but continues; WS features break later with no clear cause.
  5. Telegram is mandatory outside the wizard. User must edit .env by hand and restart - breaks the "one command" promise.
  6. 2869-line god file. Hard to maintain, hard to test, hard to extend.
  7. No resume, no progress persistence. Failure at step 7 = restart from step 1. No "you are on phase 3 of 7" tracking.
  8. Solid plumbing underneath. Secrets, migrations, ports, service install, color handling - all well done. The plumbing is not the problem; the journey is.
